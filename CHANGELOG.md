Version 1.8.12 - Duplicate-on-OS badges, [OS] title links, clearer big-file extraction errors

Three forum-requested improvements from 0pener (forum post p54971):

1) "N on OS" badge per matched subtitle row (per imdb + language)
   - New XmlRpcService.searchSubtitlesByImdb with 24h cache (de-duped by

   - New useAutoOsDuplicateCount hook that fires when (movie, language)
     settles for each pair; results fanned out to all rows sharing the key
   - MatchedPairs row, right side (ml-auto), surfaces:
       🟡 N on OS   — amber badge linking to OS search filtered by
                       imdbid + sublanguageid
       🟢 0 on OS — new language for this title — positive "fresh"
                       signal in green
       ⏳ checking OS… / ⚠️ OS check failed — transient states
   - Replaces the long-disabled "Language-specific subtitle count" stub

2) [OS] links next to every IMDB ID in the movie display
   - Two new helpers in fileUtils.js: getOsImdbSearchUrl(imdbId) and
     getOsImdbLanguageSearchUrl(imdbId, sublang)
   - Six [OS] links inserted across MovieDisplay (3 link positions × 2
     code branches: featuresData + XML-RPC fallback) — next to Episode,
     TV Series, and Movie/Main IMDB IDs
   - 0pener's complaint was "titles link to IMDB twice, but not to OS";
     now every IMDB ID has a sibling [OS] link

3) Clearer error when extraction fails on huge MKVs
   - Root cause is NOT mkvmerge version (0pener's hypothesis was a
     correlation): FileReader.readAsArrayBuffer hits a ~2GB browser
     memory cap. @opensubtitles/video-metadata-extractor's "88GB"
     chunking reassembles all chunks into one Blob, which still needs
     a single ArrayBuffer when handed to @ffmpeg/util's fetchFile()
   - New BROWSER_FILEREADER_SOFT_LIMIT (2GB) and pre-flight warning
     when file size exceeds it
   - New humanizeExtractorError(error, file) rewrites the cryptic
     "Failed to extract metadata: File could not be read! Code=-1" into:
       Browser cannot read files this large (X.XX GB). The browser's
       FileReader hits a ~2 GB memory cap regardless of the extractor's
       chunking. Workaround: extract the subtitle tracks locally with
       mkvextract or ffmpeg, then drop the .srt files onto the
       uploader directly.
   - Wrapped at all three throw sites in mkvSubtitleExtractor.js
   - Bug-fix bonus: UI's "extraction_failed" status was set but never
     rendered (badge only matched "error"). Combined them — both now
     surface as "❌ Extraction failed" with the humanized error in
     the tooltip
   - TODO comment documents the durable fix: Tauri sidecar spawning
     native mkvextract/ffmpeg (tauri-plugin-shell + filesystem already
     in place; needs binary bundling + signing per-platform)

Tests: 207/207 pass. Build clean.

## v1.8.11 - v1.8.11

*Released: 6/9/2026*

Version 1.8.11 - Fix Hearing Impaired false positives on title "Hi"

The auto-detection regex `\bhi\b` and `.hi.` were matching any "Hi" in a release
filename, falsely flagging ~500 episodes and ~100 movies as Hearing Impaired.

Forum-reported failing cases (now correctly classified as NOT HI):
  - The.Hi-Lo.Country.1998.1080p.USA.BluRay.AVC.DTS-HD.MA.5.1-rmHD
  - The.King.of.Queens.S03E13.Hi.Def.Jam.1080p.NF.WEB-DL.AAC2.0.H.264-POWER
  - For.All.Mankind.S01E07.Hi.Bob.1080p.WEB-DL.6CH.x265.HEVC-PSA

Fix in src/services/subtitleUploadService.js:
  - checkHearingImpairedFromString: dropped bare \bhi\b / hi[_-] / [_-]hi.
    Now only fires on sdh, psdh, hearing.impaired variants, or a path part
    that IS exactly "hi" (e.g. an /HI/ directory).
  - checkHearingImpairedFromFilename: switched to positional token matching.
    Strip subtitle extension, tokenize on .-_, then look for hi/sdh/psdh/
    hearingimpaired only in the LAST 3 TOKENS — the language-code / feature-
    flag slot where real HI markers live (movie.eng.hi.srt, show.sdh.srt).
    hearing.impaired keyword still matches anywhere.

Real HI markers in the language-code slot still detect correctly:
  - movie.hi.srt, movie.eng.hi.srt, movie-hi-en.srt, show.s01e01.sdh.srt

Tests: 48 pass (38 pre-existing + 10 new regression tests covering the
three forum cases and reverse-direction "must still detect" sanity checks).

- Automatically removes language codes (eng, spa, por, fra, etc.)
- Removes regional language variants (pt-PT, pt-BR, en-US, es-ES, etc.)

- Removes CD markers (CD1, CD I)
- Removes SDH/HI suffixes
- Removes addic7ed.com and other custom patterns
